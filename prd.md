프로젝트명: Cross-Environment Diff & Validation Engine
Phase 1: 아키텍처 및 데이터 흐름 설계
가장 먼저 앱의 뼈대를 잡아야 합니다. AI에게 전체적인 청사진을 먼저 이해시키는 단계입니다.

입력(INPUT) 파이프라인: 개발(Dev)과 검증(QA) 환경의 API 엔드포인트 또는 DB 쿼리 실행기에 동일한 파라미터(JSON 형식 등)를 동시에 쏘아주는 Injector 모듈.

출력(OUTPUT) 수집기: 두 환경에서 반환된 결과값(Response Body, 연산 결과, 로그 등)을 수집하여 정규화(Normalization)하는 모듈.

비교(Diffing) 엔진: 수집된 두 데이터를 노드(Node) 단위로 비교하는 핵심 엔진.

Phase 2: 핵심 기능 (Core Features) 정의
바이브 코딩 시 각 기능을 하나의 독립적인 프롬프트로 나누어 개발을 지시하면 훨씬 정교한 코드를 얻을 수 있습니다.

Strict Equality vs. Intentional Mutation (엄격한 일치 및 의도적 변경 허용)

기본적으로 Dev와 QA의 아웃풋은 100% 일치해야 합니다.

단, 이번 배포로 인해 변경이 예상되는 특정 필드(Target Fields)는 사용자가 미리 설정(Config)하여, 해당 부분의 차이점은 '오류'가 아닌 '정상 반영'으로 처리하는 로직을 구현합니다.

종속성 기반 로직 검증 (Logic Constraint Testing)

단순한 값 비교를 넘어 데이터베이스 테이블 간의 필수 관계도 검증합니다.

예시 시나리오: "결과 'C'가 도출되었을 경우, 반드시 상응하는 결과 'E'가 함께 생성되어야 한다"는 필수 종속성(Hierarchy) 규칙을 테스트 케이스에 포함하여, 두 환경 모두 이 룰을 위반하지 않는지 체크합니다.

직관적인 로그 파서 및 시각화 (Log Parser & Dashboard)

터미널에 텍스트만 뿌려주는 것을 넘어, 두 JSON 데이터 간의 차이점을 색상으로 강조(Highlight)해 주는 뷰어를 만듭니다. (예: 추가된 값은 초록색, 변경/누락된 값은 붉은색).

Phase 3: 바이브 코딩 프롬프트 로드맵
AI(Gemini, Cursor 등)에게 코딩을 시킬 때 아래 순서대로 지시하면 앱이 견고하게 완성됩니다.

Step 1 (환경 세팅): "Node.js와 JavaScript를 사용해서, 두 개의 다른 URL로 동일한 JSON Payload를 비동기적으로 전송하고 결과를 받아오는 기본 모듈을 작성해 줘."

Step 2 (비교 로직): "받아온 두 개의 JSON 객체를 깊은 비교(Deep Compare)하는 함수를 만들어 줘. 값이 다를 경우 어느 Path에서 차이가 났는지 배열로 반환해야 해."

Step 3 (예외 처리): "앞서 만든 함수에 '무시할 키값(Ignore Keys)' 배열을 인자로 추가해서, 내가 수정한 부분의 차이는 에러로 잡지 않도록 로직을 고도화해 줘."

Step 4 (UI/시각화): "비교 결과를 한눈에 볼 수 있도록 HTML/CSS/JS(또는 Firebase를 연동한 대시보드)로 간단한 웹 UI를 만들어 줘. 좌측엔 QA, 우측엔 Dev 결과를 두고 차이점을 하이라이트 해줘.
